Discovery Findings

Five pain points anchoring this engagement

What we heard during discovery, in your words — with the operational cost each one carries today.

Pain Point 1

Legacy Technical Debt

Org built 2014–2016 on Apex, Visualforce, workflow rules, and process builders. Modern features like Flow, Data Cloud, and Agentforce can't be safely adopted because so much business logic is locked in untested code paths.

Modernization is blocked: every new platform capability requires months of compatibility analysis before it can be turned on.
Pain Point 2

Pricing Across Divisions

Print runs on a custom Apex Estimator. Foam3D™, BoxPop®, and Woodworking each rely on offline tools — spreadsheets and standalone calculators — that aren't integrated with Salesforce.

Three of four divisions can't quote, version-control, or audit pricing on-platform — reps re-enter numbers in Salesforce manually after the fact.
Pain Point 3

Inflexible Catalog

Adding a new capability like fabric printing means changing Apex classes, rebuilding Visualforce pages, and running a full QA cycle. The "product catalog" is essentially hard-coded inside the Estimator.

New product variants take 6–12 weeks from business request to production — time-to-market lags well behind operational readiness.
Pain Point 4

Lean Internal Team

Two-person IT department. Strong stated preference for admin-friendly configuration. Most of the org was built and is maintained internally because the original partner engagement didn't land.

Every business request — pricing tweak, new field, workflow change — funnels through 1.5 FTE. Technical debt compounds because there's no headroom to refactor.
Pain Point 5

Order Decomposition

One quote line can represent 100 art files or shipments to multiple states. Today this is handled with custom junction objects — Fulfillment__c and Invoice_Line_Item__c — sitting between Job_Item__c and Invoice__c.

Quote vs. order vs. invoice mismatches recur with customers; the hand-built decomposition layer is brittle and edge cases need code changes.

Architectural Validation

Does Britten's current state actually map to RCA?

A Salesforce-Architect-level walkthrough: where the mapping is strong, and the risks we'd flag before any code is written.

Verdict: the current data model and process flow map cleanly to Revenue Cloud Advanced's standard objects. The shape of the business does not need to change — only the implementation underneath.

Where the mapping is strong

✓
Job__c → Order

Britten's custom Order behavior maps to RCA's standard Order, which extends Order with decomposition and orchestration.

✓
Job_Item__c (~450 fields) → Order Line + Product Attributes

The "everything is custom" attribute pattern is exactly RCA's attribute-based catalog — attributes move off the line item and onto the product definition.

✓
Fulfillment__c + Invoice_Line_Item__c → Fulfillment Order Lines

The junction layer is replaced by RCA's native decomposition: one commercial line splits into N fulfillment lines via configured rules.

✓
Apex Print Estimator → Pricing Procedure + Constraint Builder

Cost-plus and bell-curve logic is achievable declaratively. Genuinely unique math (waste %, complexity-based labor) plugs in as an isolated Apex action invoked by the Pricing Procedure.

✓
Closed-Won Apex trigger → Closed-Won Flow

The Opportunity-to-Order handoff is a standard pattern in RCA. QuoteLineItems map to Order Lines automatically.

Risks to validate before commit

⚠
Pricing parity is the #1 risk

Cost-plus + bell-curve + free rep override needs to be replayed against 12+ months of historical Estimator output before go-live. Any divergence must be explainable.

⚠
450 fields is a re-modeling exercise, not a 1:1 migration

Each custom field has to be classified: attribute on the product, attribute on the line, calculated by a procedure, or genuinely orphan. This is the bulk of Phase 1 effort.

⚠
Invoice continues to start in Salesforce, then sync to BC

RCA Billing is explicitly NOT adopted — Business Central remains the financial system of record. The existing SF→BC integration must be preserved bit-for-bit so the accounting team's workflow is undisturbed.

⚠
Manufacturing floor screens stay custom

Pre-press queue, print operator queue, and QR job sheets are out of RCA scope. They remain on Visualforce (or migrate to LWC as a separate workstream) — not bundled into this initiative.

Decisions Required

Three calls that shape the recommendation

Where Britten's input determines the path forward.

Decision 1

Pilot Division

Print first is the obvious choice (mature data, clearest pricing). After Print, which division do we tackle next: Foam, BoxPop, or Woodworking?

Decision 2

Phase 1 Pricing Test

Pick the most complex pricing rule from the Estimator. The implementation partner will model it in Constraint Builder and bring it back as proof.

Decision 3

Parity Sign-Off

Who signs off on pricing parity before go-live? We need a named accountable owner, not a role.
